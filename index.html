A multi-page Power BI analytics dashboard using AI-generated ticketing, social media, and fan survey data. Used modeled datasets in MySQL, SQL joins, and DAX measures to analyze relationships between fan engagement, revenue, and game demand — replicating how a professional sports organization might approach business intelligence.

📊  Data Creation  →  SQL Database  →  SQL Joins & Queries  →  Dashboard Design  →  Overall Insights
Step 1 Data Generation

Used AI to generate synthetic data of realistic fan datasets. Four CSV files with 100+ rows each:

fans.csv
Name · Age · City · Email · Fan_ID · Season Ticket Holder
social_metrics.csv
Date · Engagement Rate · Followers · Posts · Video Views
survey.csv
Fan_ID · Game Satisfaction · NPS · Parking Satisfaction · Comments
ticket_sales.csv
Game_ID · Attendance · Date · Opponent · Ticket Sales
Step 2 SQL Database

Datasets were created into a MySQL database to ensure all data was isolated in a structured schema. Two tables were designed to support ticketing and social analytics:

Database & Table Creation
CREATE DATABASE IF NOT EXISTS ticketing_analytics;
USE ticketing_analytics;

CREATE TABLE social_metrics (
    date             DATE,
    followers        INT,
    engagement_rate  FLOAT,
    posts            INT,
    video_views      INT
);

CREATE TABLE ticket_sales (
    game_id   INT PRIMARY KEY,
    date      DATE,
    opponent  VARCHAR(100),
    attendance INT,
    revenue   FLOAT
);
Why this Works
--> Explicit data types ensure clean modeling
--> game_id acts as a primary key
--> date serves as the relational join key between ticket and social data
--> This structure mimics how professional sports organizations store event-level data
Step 3 Data Ingestion

CSV files were directly imported into MySQL:

ETL — Load Data
LOAD DATA INFILE '.../social_metrics.csv'
INTO TABLE social_metrics
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
IGNORE 1 ROWS;

LOAD DATA INFILE '.../ticket_sales.csv'
INTO TABLE ticket_sales
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
IGNORE 1 ROWS;
What this Shows
--> Basic ETL (Extract, Transform, Load)
--> Structured data ingestion
--> Preparing analysis-ready relational tables
Step 4 SQL Joins & Queries

To analyze the relationship between digital engagement and ticket performance, a relational join was performed:

SQL Join — Engagement vs Revenue
SELECT
    t.date,
    t.opponent,
    t.attendance,
    t.revenue,
    s.engagement_rate,
    s.video_views,
    s.posts
FROM ticket_sales t
JOIN social_metrics s
    ON t.date = s.date;
What this Join Achieves
--> Aligns game-day revenue with social engagement metrics
--> Enables direct correlation analysis
--> Produces a unified dataset for Power BI modeling
--> Reduces transformation complexity inside the BI layer
Disclaimer: The Fan and Survey datasets were imported directly into Power BI as CSV files. The Social Media and Ticket Sales datasets were intentionally processed in MySQL to demonstrate SQL database design, table creation, data ingestion, and relational joins prior to BI modeling. This approach allowed me to showcase both SQL data engineering skills and Power BI analytics development within the same project.
Insights Overall Insights
--> Rivalry games drive the most revenue — 76ers and Pistons matchups generated the highest ticket revenue (~$20M each) and also had among the highest social engagement rates, making them the most valuable nights commercially and digitally
--> Social engagement and revenue move inversely — revenue peaks early then declines, while engagement rate trends upward toward season's end — the disconnect may be worth exploring for future marketing timing
--> Parking satisfaction is dragging down NPS — overall NPS of -44 is significantly negative, with a clear relationship between parking satisfaction and NPS scores across cities — the clearest area for operational improvement
--> Cambridge fans are the most satisfied with parking (3.16 avg) despite being a smaller season ticket group (19 holders) — a potential conversion opportunity

Project 02 · Python · Predictive Model

NBA Points Per
Game Predictor

A live data-driven model for predicting NBA player scoring output

↗ View on GitHub

This project uses Python to build a predictive model for NBA player Points Per Game (PPG). Rather than relying on static datasets, it scrapes live data directly from Basketball-Reference, pulling current season stats for all active players to ensure the predictions reflect the most up-to-date performance.

The model factors in both offensive metrics (PPG, usage rate, shooting efficiency) and defensive context (opponent Defensive Rating, DRtg) as well as home/away game splits — since players' performance often varies significantly by venue. This results in more nuanced and accurate predictions than a simple average-based approach.

Key Highlights

  • Scraped live player stats from Basketball-Reference using Python, keeping predictions current with real season data
  • Incorporated opponent Defensive Rating (DRtg) as a key feature to account for matchup difficulty
  • Built separate models for home and away games to capture venue-based performance differences
  • Engineered features from raw stats including shooting efficiency and usage rate to improve model accuracy
  • Designed the pipeline to be rerun easily as new game data becomes available throughout the season
